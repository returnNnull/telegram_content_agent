# Спецификация Telegram Content Agent

## 1. Обзор

Система анализирует изменения в репозитории, находит инженерные решения, которыми имеет смысл делиться публично, и превращает их в черновик поста для Telegram. Первая версия должна быть не полностью автономным издателем, а управляемым контентным пайплайном с обязательным подтверждением человеком.

## 2. Формулировка проблемы

Команды регулярно внедряют технически значимые улучшения, но эти изменения остаются внутри коммитов, пул-реквестов и рабочих обсуждений. Подготовка публичного технического поста вручную стоит дорого, потому что требует отдельно собирать фактуру и отдельно превращать ее в внятный рассказ.

Цель проекта в том, чтобы сократить эти затраты и превратить изменения в кодовой базе в структурированный вход для контентного процесса.

## 3. Цели

- находить значимые технические изменения в истории репозитория
- отделять сигнал от шума до этапа генерации текста
- объяснять изменения так, чтобы они были полезны внешней инженерной аудитории
- собирать Telegram-черновики с проверяемой фактурой
- сохранять человека в контуре перед публикацией

## 4. Не-цели

- публиковать каждый коммит или каждый влитый пул-реквест
- заменять заметки к релизу или внутренний журнал изменений
- делать полностью автономную публикацию в MVP
- генерировать маркетинговые тексты вместо технических
- поддерживать сразу несколько контентных каналов в первой версии

## 5. Пользователи

### Основной пользователь

Евангелист разработки, основатель или инженер, который хочет регулярно публиковать технические обновления, основанные на реальной работе команды.

### Вторичный пользователь

Редактор или ревьюер, который проверяет качество текста, тон и отсутствие чувствительных деталей перед публикацией.

## 6. Поток верхнего уровня

```text
Источник репозитория / пул-реквестов
  -> Change Scanner
  -> Change Structurer
  -> Interestingness Scorer
  -> Topic Selector
  -> Draft Writer
  -> Human Review
  -> Telegram Publisher
```

## 7. Архитектура

### 7.1 Модель исполнения

Для MVP предпочтителен один сервис с явными стадиями пайплайна. Мультиагентную модель на первом этапе лучше симулировать внутренними модулями со строгими контрактами данных между шагами. Это упрощает разработку и эксплуатацию, но сохраняет возможность позже выделить отдельные сервисы или агентов.

### 7.2 Режимы запуска

- ручной запуск для выбранного диапазона дат или ревизий
- запуск по расписанию, например раз в неделю
- событийный запуск от влитых пул-реквестов на следующих итерациях

### 7.3 Источники истины

- git history
- метаданные пул-реквестов
- манифесты зависимостей и lockfile-файлы
- CI-конфигурация и инфраструктурные файлы
- опционально комментарии ревьюера и история прошлых публикаций

## 8. Компоненты

### 8.1 Change Scanner

Зона ответственности:
- читать изменения в репозитории за выбранный период или диапазон ревизий
- извлекать коммиты, измененные файлы, статистику diff и сырые фрагменты diff
- подтягивать ссылки на пул-реквесты, если они доступны

Вход:
- путь к репозиторию
- временной диапазон или диапазон ревизий

Выход:
- сырые события изменений

Примечания к реализации:
- на старте достаточно `git log`, `git diff` и файловой статистики
- лучше дробить diff по файлам или чанкам, чем отправлять модели весь diff целиком

### 8.2 Change Structurer

Зона ответственности:
- превращать сырые изменения в нормализованные сущности
- классифицировать тип изменения и домен
- определять технологии, паттерны и вероятный инженерный смысл

Рекомендуемые категории:
- feature
- bugfix
- refactor
- performance
- infra
- deps
- tests
- docs

Рекомендуемые домены:
- backend
- frontend
- data
- ci_cd
- platform
- product_logic

Выход:
- структурированные события изменений

### 8.3 Interestingness Scorer

Зона ответственности:
- оценивать, стоит ли превращать изменение в публичный технический пост
- объяснять, почему тема может быть полезна внешнему читателю

Измерения скоринга:
- novelty
- impact
- explainability
- uniqueness

Сильные сигналы:
- новая инфраструктура или интеграция внешнего сервиса
- содержательный refactor с понятной архитектурной причиной
- улучшение производительности или надежности
- queueing, caching, observability, retries, auth, rate limiting
- миграция между инструментами или подходами

Слабые сигналы:
- форматирование без изменения поведения
- малозначимые обновления зависимостей
- перенос файлов без функциональных изменений
- рутинные bugfix без переносимой инженерной идеи

### 8.4 Topic Selector

Зона ответственности:
- ранжировать кандидатов и выбирать от одной до трех тем на пост
- удалять пересекающиеся темы
- балансировать между пользой для аудитории и технической глубиной

Правила отбора:
- избегать почти одинаковых тем в одном прогоне
- избегать тем, которые недавно уже публиковались
- предпочитать темы с хорошей фактурой и понятной ценностью

### 8.5 Draft Writer

Зона ответственности:
- собирать черновик, готовый для Telegram, на основе структурированных данных
- оставаться фактическим и опираться на подтверждающую фактуру
- соблюдать заданный редакционный стиль

Рекомендуемая структура черновика:
- `headline` - заголовок
- `lead` - короткий лид
- `what changed` - что изменилось
- `why it is interesting` - почему это интересно
- `technologies and patterns used` - какие технологии и паттерны использовались
- `outcome or tradeoff` - какой получился результат или компромисс
- `closing line` - короткое завершение

### 8.6 Human Review

Зона ответственности:
- одобрять, отклонять или отправлять на доработку
- ловить конфиденциальные детали, завышенные утверждения и слабые объяснения

Чеклист ревью:
- текст фактически корректен
- текст безопасно публиковать
- есть понятная ценность для внешнего читателя
- текст достаточно короткий для Telegram
- тема не повторяет недавние публикации

### 8.7 Telegram Publisher

Зона ответственности:
- публиковать одобренные черновики через Telegram Bot API
- сохранять метаданные публикации
- помечать историю тем, чтобы не повторяться

## 9. Контракты данных

### 9.1 RawChangeEvent

```json
{
  "id": "commit:abc123",
  "source_type": "commit",
  "title": "Добавить политику ретраев для фоновых задач",
  "author": "Jane Doe",
  "timestamp": "2026-03-10T12:34:56Z",
  "files": [
    "src/jobs/worker.ts",
    "src/queue/retry-policy.ts"
  ],
  "insertions": 84,
  "deletions": 17,
  "diff_summary": "Добавлены экспоненциальный backoff и идемпотентная обработка повторных запусков задач",
  "references": {
    "pr_url": "https://example.com/pr/123"
  }
}
```

### 9.2 StructuredChangeEvent

```json
{
  "id": "commit:abc123",
  "category": "infra",
  "domain": "backend",
  "plain_summary": "Фоновые задачи теперь повторяются безопасно и не падают навсегда после временной ошибки",
  "technologies": ["BullMQ", "Redis"],
  "patterns": ["retry", "backoff", "idempotency"],
  "evidence": [
    {
      "file": "src/queue/retry-policy.ts",
      "reason": "Появилась отдельная абстракция политики ретраев"
    }
  ],
  "risk_level": "medium"
}
```

### 9.3 ScoredTopic

```json
{
  "id": "topic:retry-policy",
  "change_ids": ["commit:abc123"],
  "score": 0.84,
  "novelty": 0.8,
  "impact": 0.9,
  "explainability": 0.85,
  "uniqueness": 0.8,
  "why_interesting": "Показывает, как команда повысила надежность фоновых задач за счет retry и idempotency вместо ручного восстановления",
  "reject_reason": null
}
```

### 9.4 ArticleDraft

```json
{
  "id": "draft:2026-03-11-retry-policy",
  "topic_id": "topic:retry-policy",
  "headline": "Как мы сделали фоновые задачи устойчивее к ошибкам",
  "lead": "Недавнее изменение в backend добавило контролируемые ретраи и backoff для упавших задач.",
  "sections": [
    {
      "title": "Что изменили",
      "content": "В пайплайне обработки задач появились повторные попытки для временных ошибок с явной стратегией backoff."
    },
    {
      "title": "Почему это важно",
      "content": "Это снижает объем ручного восстановления и делает асинхронную обработку заметно надежнее."
    }
  ],
  "telegram_text": "Как мы сделали фоновые задачи устойчивее к ошибкам ...",
  "evidence": [
    {
      "type": "commit",
      "id": "commit:abc123"
    }
  ],
  "status": "draft"
}
```

## 10. Контракты промптов

### 10.1 Prompt для структурирования

```text
Ты анализируешь изменения в репозитории для извлечения технических тем.
Верни строгий JSON со следующими полями:
- category
- domain
- technologies
- patterns
- plain_summary
- evidence

Правила:
- игнорируй форматирование и низкоинформативные изменения
- не выдумывай детали реализации
- пиши кратко и простым языком
```

### 10.2 Prompt для скоринга

```text
Ты оцениваешь, подходит ли изменение в репозитории для публичного технического поста в Telegram.
Верни строгий JSON со следующими полями:
- score (0..1)
- novelty
- impact
- explainability
- uniqueness
- why_interesting
- reject_reason

Отклоняй темы со слабой образовательной ценностью или те, которые невозможно понять без скрытого внутреннего контекста.
```

### 10.3 Prompt для написания текста

```text
Напиши черновик поста для Telegram-канала с инженерной аудиторией.

Ограничения:
- 800-1500 символов
- без маркетинговой воды
- только факты, подтвержденные входными данными
- объясни инженерную причину и ценность изменения
- если фактуры недостаточно, укажи это вместо догадок
```

## 11. Хранилище

### 11.1 Состояние, которое нужно хранить

- уже обработанные диапазоны ревизий
- историю тем
- историю черновиков
- решения ревью
- метаданные опубликованных постов

### 11.2 Предлагаемая схема

Таблицы или коллекции:
- `runs`
- `raw_changes`
- `structured_changes`
- `topics`
- `drafts`
- `reviews`
- `publications`

Для MVP достаточно SQLite. Переход на Postgres имеет смысл только при нескольких воркерах или отдельной операционной обвязке.

## 12. Фильтрация и защитные ограничения

### 12.1 Фильтры шума

Нужно отбрасывать или сильно понижать в рейтинге:
- коммиты только с форматированием
- сгенерированные файлы
- большие изменения только в lockfile-файлах
- массовые переименования без изменения поведения
- обновление версии без заметного содержательного эффекта

### 12.2 Фильтры безопасности

Публикация должна блокироваться, если в черновике есть:
- приватные URL или внутренние имена сервисов
- учетные данные или чувствительные идентификаторы
- детали дорожной карты, не предназначенные для публикации
- непроверяемые утверждения

### 12.3 Правила против галлюцинаций

- каждое публичное утверждение должно опираться на одно или несколько подтверждений
- модуль генерации текста должен работать только с нормализованными входами, а не с неограниченным сырым контекстом
- если фактуры не хватает, тема должна терять оценку или блокироваться

## 13. Оценка качества

### 13.1 Функциональные метрики

- число кандидатных тем за прогон
- доля выбранных тем, которые проходят ручное ревью
- доля черновиков, публикуемых без тяжелого переписывания

### 13.2 Качественные метрики

- оценка ревьюером фактической точности
- оценка ревьюером пользы для внешнего читателя
- частота повторения похожих тем
- расстояние редактирования между исходным и одобренным черновиком

## 14. Бэклог MVP

### Итерация 1

Цель:
Получать локальный markdown-черновик из недельного диапазона git-изменений.

Состав работ:
- локальное сканирование репозитория
- суммаризация diff
- скоринг тем
- выбор от одной до трех лучших тем
- генерация markdown-черновика

Результат:
- CLI-команда
- JSON-артефакты
- markdown-черновик

### Итерация 2

Цель:
Улучшить качество тем и добавить память о прошлых публикациях.

Состав работ:
- интеграция с метаданными PR
- хранение истории тем
- защита от повторов
- более сильная фильтрация шума
- настраиваемый редакционный тон

### Итерация 3

Цель:
Добавить процесс ревью и публикации.

Состав работ:
- интеграция с Telegram Bot API
- конечный автомат для `draft -> approve -> publish`
- логи публикации
- базовый интерфейс ревьюера или поток согласования через отдельный чат

## 15. Рекомендуемый стек для MVP

- язык: Python или TypeScript
- доступ к репозиторию: сначала git CLI, затем VCS API
- LLM-интеграция: OpenAI API со строгими JSON-ответами
- хранилище: SQLite
- доставка: Telegram Bot API
- расписание: cron или легкий планировщик задач

## 17. Открытые вопросы

- Что считать основной единицей анализа: влитые PR, коммиты или диапазоны между релизами?
- Какой тон нужен каналу: глубоко инженерный, от лица основателя или смешанный?
- Какой объем ручного редактирования допустим перед публикацией?
- Нужны ли публичные ссылки на commit и PR прямо в посте, или их достаточно хранить только в метаданных для ревьюера?

## 18. Следующий шаг в реализации

Первой прикладной целью должна быть CLI-команда, которая принимает диапазон дат, извлекает изменения из репозитория, строит структурированных кандидатов в темы и сохраняет один markdown-черновик вместе с JSON-артефактами для ручной проверки.
