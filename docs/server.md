# Сервер публикации в Telegram

## Назначение

Сервер принимает структурированные данные поста и публикует их в Telegram-канал через Bot API.

Основная идея:
- модель или внешний агент готовит содержимое поста
- сервер отвечает только за доставку в Telegram
- логика отправки картинок и ссылок не размазывается по промптам

## Контракт запроса

`POST /publish`

Требует заголовок:

```text
Authorization: Bearer <PUBLISH_API_TOKEN>
```

```json
{
  "text": "<b>Заголовок</b>\n\nТекст поста",
  "parse_mode": "HTML",
  "images": [
    {
      "url": "https://example.com/image.png"
    },
    {
      "path": "/absolute/path/to/image.png"
    }
  ],
  "links": [
    {
      "title": "PR",
      "url": "https://github.com/org/repo/pull/123"
    }
  ],
  "link_style": "buttons",
  "disable_web_page_preview": false,
  "disable_notification": false,
  "protect_content": false,
  "dry_run": false,
  "chat_id": "@optional_override"
}
```

## Как сервер выбирает стратегию публикации

- `0` картинок: `sendMessage`
- `1` картинка: `sendPhoto`
- `2-10` картинок: `sendMediaGroup`

Если caption не помещается в лимит Telegram:
- сначала уходят картинки
- потом отдельным сообщением уходит текст

Если `link_style = "buttons"`:
- ссылки отправляются как inline-кнопки

Если `link_style = "text"`:
- ссылки добавляются в конец текста поста

## Ограничения

- сервер сейчас поддерживает только `HTML` parse mode или текст без parse mode
- `sendMediaGroup` не умеет inline-кнопки, поэтому при нескольких картинках кнопки уйдут отдельным сообщением
- для локальных изображений нужно передавать абсолютный путь
- за один запрос допускается до `10` картинок
- endpoint публикации защищен bearer-токеном и должен вызываться только доверенным клиентом

## Рекомендуемый сценарий использования

1. Агент анализирует изменения в репозитории
2. Агент собирает `article_draft.json`
3. Клиент или CLI вызывает `POST /publish`
4. Сервер публикует данные в Telegram
