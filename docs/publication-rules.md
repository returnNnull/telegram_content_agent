# Правила публикации

## Роли в системе

- Внешний агент или скрипт готовит payload и отправляет его в `POST /drafts`.
- `moderation bot` публикует preview поста в `MODERATION_CHAT_ID` и показывает control message со статусом и кнопками.
- Модератор в review-группе принимает решение: публиковать сразу, запланировать или отклонить.
- `publisher bot` публикует подтвержденный пост в `TELEGRAM_CHANNEL_ID`.
- Scheduler исполняет отложенные публикации и синхронизирует статус moderation draft-а после фактической отправки.

## Штатный путь публикации

- Основной endpoint сервиса: `POST /drafts`.
- `POST /publish` и `POST /schedule` считаются bypass-сценариями. Используйте их только для ручного форса, аварийного обхода moderation flow или технической проверки.
- В `POST /drafts` поле `chat_id` игнорируется. Финальная публикация всегда идет в канал из `TELEGRAM_CHANNEL_ID`.
- В `POST /drafts` нельзя передавать `dry_run`.
- Локальные статьи из `ARTICLES_ROOT_PATH/**/articles/*.md` автоматически подхватываются сервисом при старте и отправляются на модерацию, если еще не проходили этот цикл.

## Правила подготовки payload

- В запросе должен быть хотя бы один источник контента: `text`, `images` или `links`.
- HTML-текст ограничен `4096` символами.
- Plain-text без `parse_mode` может быть длиннее `4096`: сервис разобьет его на несколько сообщений.
- Можно передавать до `10` изображений и до `10` ссылок.
- Заголовок кнопки ссылки ограничен `64` символами.
- Для локальных изображений нужно передавать абсолютный путь, доступный на сервере.
- `link_style = "buttons"` рендерит ссылки inline-кнопками.
- `link_style = "text"` добавляет ссылки в конец текста поста.
- Если используется `link_style = "text"`, оставляйте запас по длине текста под URL, потому что ссылки дописываются после валидации поля `text`.
- Если у поста одна картинка и caption длиннее `1024` символов, сервис отправит картинку и отдельное текстовое сообщение.
- Если у поста несколько картинок, используется `sendMediaGroup`. Inline-кнопки у media group не поддерживаются, поэтому при `link_style = "buttons"` сервис отправит ссылки отдельным сообщением.
- Если текста нет, а есть только ссылки, сервис добавит служебный текст `Полезные ссылки`.

## Правила модерации

- Решения по draft-ам принимаются только в `MODERATION_CHAT_ID`.
- Если задан `MODERATION_ALLOWED_USER_IDS`, только эти Telegram user id могут нажимать inline-кнопки, отправлять время публикации и комментарии к отклонению.
- Если `MODERATION_ALLOWED_USER_IDS` пуст, модерировать может любой участник review-группы.
- У draft-а в статусе `pending_review` доступны кнопки `Опубликовать сейчас`, `Запланировать` и `Отклонить`.
- У draft-а в статусе `awaiting_schedule` доступны `Опубликовать сейчас`, `Отменить ввод времени` и `Отклонить`.
- У draft-а в статусе `awaiting_rejection_comment` доступна кнопка `Отменить отклонение`.
- Если немедленная публикация в канал завершается ошибкой, сервис показывает ошибку модератору через callback и не переводит draft в `failed`. Повторить действие можно с того же control message.

## Правила планирования времени

- После нажатия `Запланировать` draft переходит в `awaiting_schedule`.
- Время должен прислать отдельным сообщением тот же модератор, который нажал `Запланировать`.
- Поддерживаются форматы `YYYY-MM-DD HH:MM`, `DD.MM.YYYY HH:MM`, `HH:MM` и ISO 8601.
- Если timezone в сообщении не указана, сервис использует `MODERATION_TIMEZONE`.
- Если передан только формат `HH:MM` и это время уже прошло сегодня в `MODERATION_TIMEZONE`, сервис переносит публикацию на следующий день.
- Сообщение `/cancel` отменяет ожидание времени и возвращает draft в `pending_review`.

## Правила отклонения

- После нажатия `Отклонить` draft переходит в `awaiting_rejection_comment`.
- Комментарий к отклонению должен прийти отдельным сообщением от того же модератора.
- Сообщение `/cancel` отменяет ввод комментария и возвращает draft в `pending_review`.
- Комментарий сохраняется в SQLite и доступен через публичный HTTP API.
- Для автоправки markdown-статьи на следующем старте можно использовать инструкции:
  - `title: Новый заголовок`
  - `replace: старый текст => новый текст`
  - `delete: фрагмент`
  - `append: новый абзац`
  - `prepend: вводный абзац`

## Статусы moderation draft

- `pending_review` — черновик доставлен в review-группу и ждет решения.
- `awaiting_schedule` — сервис ждет время публикации отдельным сообщением.
- `awaiting_rejection_comment` — сервис ждет комментарий к отклонению отдельным сообщением.
- `scheduled` — создана запись в scheduler, ожидается фактическая отправка.
- `published` — пост отправлен в основной канал.
- `rejected` — черновик отклонен модератором.
- `failed` — не удалось доставить draft в review-группу или scheduler исчерпал попытки публикации.

## Правила scheduler

- Scheduler забирает due-задачи из SQLite и публикует их через `publisher bot`.
- При ошибке публикации scheduler повторяет попытку через `SCHEDULER_RETRY_DELAY_SECONDS`.
- Максимум попыток задается `SCHEDULER_MAX_ATTEMPTS`.
- После исчерпания попыток scheduled post получает статус `failed`, а связанный moderation draft тоже переводится в `failed`.
- `DELETE /schedule/{id}` работает только для scheduled post в статусах `pending` и `failed`.
- Если отменить scheduled post через API, связанный moderation draft не синхронизируется обратно из `scheduled`. Для draft-ов из moderation flow не рассчитывайте на этот endpoint как на пользовательскую кнопку отмены.

## Практические правила для внешнего агента

- Перед отправкой нового draft-а проверьте, что пост не дублирует уже опубликованный материал.
- В пост выносите только значимые продуктовые, инженерные и архитектурные изменения.
- По умолчанию прикладывайте ссылки на PR, спецификации, документацию или другие первичные артефакты изменения.
- Если нужен один пост с одной картинкой, старайтесь уложить текст в caption, чтобы Telegram не разбивал публикацию на фото и отдельное сообщение.
- Не закладывайтесь на прямой publish в обход review-группы как на основной процесс.
